## Phase 2: デモ & お披露目

### デモに必要な最小構成

お披露目時に **動くもの** として見せるために必要な3点セット：

1. **bsaf-jma-bot** — 実際に気象庁データを監視し、地震情報等をBSAFタグ付きでBlueskyに自動投稿している状態
2. **kazahana** — Bot定義JSONインポート → フィルタ設定画面の動的生成 → タグベースフィルタリングが動作する状態
3. **Bot定義JSON** — bsaf-jma-bot用のJSONが公開URLでホストされている状態

この3点で「プロトコル仕様書 → Bot → クライアント」の **一気通貫デモ** が成立する。

### デモシナリオ（想定）

```
1. Bot定義JSONのURLをkazahanaにインポート
   → kazahanaがJSONをパースし、Bot情報を表示
   → Botアカウントを自動フォロー
   → フィルタ設定画面が動的に生成される

2. フィルタ設定画面でユーザーが好みを設定
   → type: earthquake ✅, tsunami ✅, eruption ❌
   → value: 震度3以上 ✅
   → target: jp-kanto ✅, jp-hokkaido ✅

3. 地震が発生（または過去データでシミュレーション）
   → bsaf-jma-botがBSAFタグ付きで投稿
   → kazahanaがタグを読み取り、フィルタルールに照合
   → ユーザー設定に合致する投稿のみ表示
```

### アプローチ戦略

**方針: 「まず動くデモを見せてフィードバックをもらう」**

いきなり「引き取ってほしい」ではなく、以下のステップで進める：

1. **動作デモの提示** — BSAFの仕組みと実動するBot+クライアントを見せる
2. **フィードバック収集** — Bluesky公式の視点からの意見・課題をもらう
3. **次のステップを判断** — 反応に応じてエコシステム統合の可能性を探る

### BSAFがBluesky公式にとって魅力的な理由

- Blueskyは2026年に **「フィードをアプリの中心的な存在にする」** 方針を表明（出典: https://bsky.social/about/blog/01-28-2026-bluesky-2026-predictions-jp）
- BSAFはAT Protocolの `tags` フィールドを活用した実践的なユースケース
- サーバーサイド不要のクライアントサイドフィルタリングで、エコシステムに負荷をかけない
- ドメイン非依存設計により、防災だけでなくスポーツ・ニュース等に展開可能

### 長期的な課題意識

個人が趣味で運営し続けるには以下の点で規模が大きい：

- **プロトコルの維持** — 仕様のバージョニング、コミュニティからのPRレビュー、国際展開
- **リファレンスBotの運用** — 24/365の可用性、災害時の信頼性責任
- **kazahanaへのBSAF統合** — クライアント側の継続的メンテナンス

特に防災情報Botの運用者には「あのBotが動いていなかったせいで情報が届かなかった」という社会的責任が暗黙的に発生しうるため、個人の趣味レベルでは受け止めきれないリスクがある。

デモの反応次第では：
- BSAFプロトコル仕様をBluesky公式 or AT Protocol Foundationに移管提案
- リファレンスBotの運用をコミュニティまたは公式に移管
- 仕様の原著者・リファレンス実装者として関わり続ける形

## 決定ログ（2026-02-24セッション）

本ロードマップの基となった主要な設計決定の記録：

| 決定事項 | 結論 | 理由 |
|:---------|:-----|:-----|
| `severity` → `value` | 採用 | ドメイン非依存化。「深刻度」は防災固有の概念 |
| `country`/`region` → 1タグ統合 | 採用 | タグ枠の節約とシンプル化 |
| 統合タグの名称 | `target` | `area` は地理的制限を暗示。`target` は「誰に向けた情報か」を汎用表現 |
| `time:` タグ新設 | 採用 | 重複検知のキーとして不可欠。ソースイベント時刻を使用 |
| `source:` の扱い | 必須に昇格 | 出典の保証はプロトコルの信頼性の根幹 |
| 投稿本文へのソース記載 | 必須（二重保証） | 非BSAFクライアントユーザーへの配慮 |
| 同一ソースBotのvalue/target統一 | MUST | 重複検知の破綻を防止 |
| お披露目のアプローチ | デモ先行 | 「動くもの」を見せてからフィードバックをもらう |
