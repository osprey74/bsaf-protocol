## 1. プロジェクト概要

### 1.1 目的

気象庁の公開データ（XML PULL型フィード）を監視し、地震・津波・気象警報等の防災情報をBSAFタグ付きでBlueskyに自動投稿するBot。

### 1.2 ポジショニング

- BSAFプロトコルのリファレンスBot（最初の実装）
- kazahana（Blueskyデスクトップクライアント）のBSAFフィルタリング機能と連携

### 1.3 リポジトリ

https://github.com/osprey74/bsaf-jma-bot

## 2. 技術スタック

| レイヤー | 技術 | 理由 |
|:---------|:-----|:-----|
| 言語 | TypeScript | kazahanaと同一、@atproto/api SDKがTS向け |
| ランタイム | Node.js 24+ | LTS安定版 |
| Bluesky SDK | @atproto/api | 公式SDK、RichText・facets対応 |

| コンテナ | Docker | Fly.ioデプロイ用 |

## 3. データソース：気象庁 XML PULL型

### 3.1 フィード一覧

| URL | 内容 | 更新 | Bot監視対象 |
|:----|:-----|:-----|:-----------|
| `https://www.data.jma.go.jp/developer/xml/feed/eqvol.xml` | 地震・火山 | 毎分 | ✅ 最優先 |
| `https://www.data.jma.go.jp/developer/xml/feed/extra.xml` | 気象警報・随時 | 毎分 | ✅ 優先 |
| `https://www.data.jma.go.jp/developer/xml/feed/eqvol_l.xml` | 地震・火山（長期） | 毎時 | ✅ 起動時キャッチアップ |
| `https://www.data.jma.go.jp/developer/xml/feed/extra_l.xml` | 気象警報（長期） | 毎時 | ✅ 起動時キャッチアップ |
| `https://www.data.jma.go.jp/developer/xml/feed/regular.xml` | 定時予報 | 毎時 | ❌ 対象外（Phase 1） |
| `https://www.data.jma.go.jp/developer/xml/feed/other.xml` | その他 | 毎時 | ❌ 対象外（Phase 1） |

### 3.2 技術的制約

| 項目 | 値 |
|:-----|:---|
| ポーリング間隔 | 45秒（推奨30-60秒の中間値） |
| CDNキャッシュラグ | 毎分約20秒 |
| ダウンロード上限 | 10GB/日（超過でIPブロック） |
| プロトコル | HTTPS必須（HSTS） |
| データ形式 | RFC 4287 Atom XML |
| サービス保証 | なし（メンテナンス・遅延の可能性あり） |

### 3.3 Atomフィードのデータ構造

```xml
<feed>
  <entry>
    <id>urn:uuid:xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx</id>
    <title>震源・震度に関する情報</title>
    <updated>2026-02-15T02:55:54Z</updated>
    <link href="https://www.data.jma.go.jp/developer/xml/data/xxxxxxxx.xml" />
    <content>15日11時52分ころ、地震がありました。</content>
  </entry>
</feed>
```

- `entry.id` → 重複排除キー
- `entry.link.href` → 詳細XML取得URL（構造化データ含む）
- `entry.content` → 投稿テキストのベースに使用可能

### 3.4 詳細XMLの取得判断

すべてのエントリに対して詳細XMLを取得すると通信量が膨大になるため、以下のルールで判断する。

| 電文タイプ | 詳細XML取得 | 理由 |
|:-----------|:-----------|:-----|
| 震度速報 | ✅ 取得 | 震度・地域の構造化データが必要 |
| 震源・震度情報 | ✅ 取得 | マグニチュード・深さ・詳細震度が必要 |
| 津波警報・注意報 | ✅ 取得 | 予想津波高・到達予想時刻が必要 |
| 噴火速報 | ✅ 取得 | 火山名・噴火時刻が必要 |
| 気象特別警報 | ✅ 取得 | 対象地域の詳細が必要 |
| 気象警報・注意報 | ❌ content使用 | entry.contentで十分（通信量削減） |
| 竜巻注意情報 | ❌ content使用 | entry.contentで十分 |
| 降灰予報 | ❌ content使用 | entry.contentで十分 |
| その他 | ❌ content使用 | entry.contentで十分 |

## 5. 投稿フォーマット

### 5.1 テンプレート設計

投稿テキストはBlueskyの**300文字制限（グラフェム単位）**を遵守する。

**⚠️ 全テンプレートに `(情報源: 気象庁)` を含めること（BSAF二重出典保証の要件）。**

#### 地震情報

```
🔴 地震情報
{日時}ころ、{震源地}で地震
マグニチュード：{M値}（深さ約{深さ}km）
最大震度：{震度}
{津波の有無}
(情報源: 気象庁)
```

例：

```
🔴 地震情報
15日11時52分ころ、茨城県南部で地震
マグニチュード：5.2（深さ約50km）
最大震度：5強
津波の心配なし
(情報源: 気象庁)
```

#### 津波警報

```
🟣 【大津波警報】
{発表日時}
{対象沿岸名}
予想津波高：{高さ}
予想到達時刻：{時刻}
ただちに高台へ避難してください
(情報源: 気象庁)
```

#### 噴火速報

```
🟠 噴火速報
{火山名}で噴火が発生
発生日時：{日時}
{補足情報}
(情報源: 気象庁)
```

#### 気象特別警報

```
🟣 【特別警報】{種別}
{発表日時}
対象地域：{地域名}
{概要}
命を守る行動をとってください
(情報源: 気象庁)
```

#### 気象警報

```
🟡 気象警報
{発表日時}
{地域名}に{警報種別}
{概要}
(情報源: 気象庁)
```

#### その他（土砂災害・竜巻・記録的大雨）

```
🔶 {電文名}
{発表日時}
{地域名}
{概要}
(情報源: 気象庁)
```

### 5.2 絵文字による視覚的分類

| 絵文字 | 用途 | 対応する `value` 例 |
|:-------|:-----|:-------------------|
| 🟣 | 最高レベル（大津波警報・特別警報・南海トラフ臨時） | `special-warning`, `major-warning` |
| 🔴 | 重大（震度5以上・津波注意報・噴火） | `5-`〜`7`, `warning` |
| 🟠 | 警戒（震度3-4・火山警報・土砂災害） | `3`, `4`, `severe-warning` |
| 🟡 | 注意（震度1-2・気象警報・竜巻注意） | `1`, `2`, `warning` |
| 🔵 | 情報提供（予報解除・経過情報） | `info` |

### 5.3 投稿の言語設定

AT Protocolの `langs` フィールドを使用して言語を明示する。

```json
{
  "langs": ["ja"]
}
```

Phase 2以降で英語併記を検討する場合は `["ja", "en"]` とする。

## 6. BSAFタグ仕様（Bot投稿時）

### 6.1 タグの組み立てルール

AT Protocolの制約：**最大8タグ、合計640バイト**

全6タグが必須。以下の順でタグを付与する。

| 順序 | タグ | 例 | 必須 |
|:-----|:-----|:---|:-----|
| 1 | `bsaf:v1` | — | ✅ |
| 2 | `type:{種別}` | `type:earthquake` | ✅ |
| 3 | `value:{規模}` | `value:震度5強` | ✅ |
| 4 | `time:{ISO8601}` | `time:2026-02-15T02:52:00Z` | ✅ |
| 5 | `target:{対象}` | `target:jp-kanto` | ✅ |
| 6 | `source:jma` | — | ✅ |
| 7 | （予備1） | — | 任意 |
| 8 | （予備2） | — | 任意 |

**⚠️ `time:` は気象庁XMLの発表日時（Botの投稿時刻ではない）を使用すること。これにより複数Botからの重複検知が正しく機能する。**

### 6.2 `type` 値の一覧

| type値 | 対応する電文 |
|:-------|:-----------|
| `earthquake` | 震度速報、震源・震度情報 |
| `tsunami` | 津波警報・注意報・予報、大津波警報、津波情報 |
| `eruption` | 噴火速報、噴火警報・予報 |
| `ashfall` | 降灰予報 |
| `weather-warning` | 気象警報・注意報 |
| `special-warning` | 気象特別警報 |
| `landslide-warning` | 土砂災害警戒情報 |
| `tornado-warning` | 竜巻注意情報 |
| `heavy-rain` | 記録的短時間大雨情報 |
| `nankai-trough` | 南海トラフ地震臨時情報 |

### 6.3 `value` 値の一覧

#### 地震（気象庁 XMLのネイティブ値をそのまま使用）

| value値 | 意味 |
|:--------|:-----|
| `1` | 震度1 |
| `2` | 震度2 |
| `3` | 震度3 |
| `4` | 震度4 |
| `5-` | 震度5弱 |
| `5+` | 震度5強 |
| `6-` | 震度6弱 |
| `6+` | 震度6強 |
| `7` | 震度7 |

#### 津波

| value値 | 意味 |
|:--------|:-----|
| `advisory` | 津波注意報 |
| `warning` | 津波警報 |
| `major-warning` | 大津波警報 |

#### 気象・その他

| value値 | 意味 |
|:--------|:-----|
| `info` | 情報提供・解除 |
| `advisory` | 注意報 |
| `warning` | 警報 |
| `severe-warning` | — |
| `special-warning` | 特別警報 |

### 6.4 `target` 値の一覧

気象庁の地域区分に `jp-` プレフィックスを付けた形式を使用する。

**⚠️ 同一ソースの他のBSAF Botが存在する場合、必ず同一の `target` 値を使用すること（同一ソースBot間の相互運用性要件）。**

| target値 | 対応地域 |
|:---------|:---------|
| `jp-hokkaido` | 北海道 |

| `jp-kanto` | 関東（茨城・栃木・群馬・埼玉・千葉・東京・神奈川） |
| `jp-hokuriku` | 北陸（新潟・富山・石川・福井） |
| `jp-chubu` | 中部（山梨・長野・岐阜・静岡・愛知） |
| `jp-kinki` | 近畿（三重・滋賀・京都・大阪・兵庫・奈良・和歌山） |
| `jp-chugoku` | 中国（鳥取・島根・岡山・広島・山口） |
| `jp-shikoku` | 四国（徳島・香川・愛媛・高知） |
| `jp-kyushu` | 九州（福岡・佐賀・長崎・熊本・大分・宮崎・鹿児島） |
| `jp-okinawa` | 沖縄 |

> **Note:** 旧仕様の `region:` + `subregion:` は廃止され、`target:` に統合された。都道府県レベルのサブリージョンは、将来の拡張で `target:jp-kanto-ibaraki` のようなフォーマットで対応する可能性がある。

## 7. 投稿頻度と制御

### 7.1 Bluesky レート制限

| 制限 | 値 | 出典 |
|:-----|:---|:-----|
| レコード作成（投稿含む） | 5,000ポイント/時、35,000ポイント/日 | Rate Limits |
| 投稿1件 | 3ポイント | 同上 |
| 1時間の投稿上限 | 約1,666件/時 | 5,000 ÷ 3 |
| 1日の投稿上限 | 約11,666件/日 | 35,000 ÷ 3 |
| ログイン | IPベースで厳しい制限あり | セッション永続化で回避 |

### 7.2 想定投稿量

| シナリオ | 推定投稿数/日 | レート制限に対する割合 |
|:---------|:-------------|:---------------------|
| 平常日（有感地震2-3回、注意報数件） | 10-30件 | 0.1-0.3% |
| やや活発（群発地震、台風接近） | 50-100件 | 0.4-0.9% |
| 大規模災害（M7級＋津波＋余震多数） | 200-500件 | 1.7-4.3% |
| 最悪ケース（複合災害） | 500-1,000件 | 4.3-8.6% |

結論：レート制限に抵触する可能性は極めて低い。

### 7.3 投稿間隔ルール

| ルール | 値 | 理由 |
|:-------|:---|:-----|
| 最小投稿間隔 | 10秒 | タイムライン氾濫防止 |
| 同一電文の再投稿抑制 | entry.idで重複チェック | 二重投稿防止 |
| 投稿キュー | FIFO方式 | 複数アラートが同時発生した場合の順序制御 |
| 優先キュー | 津波・特別警報は最優先で即座に投稿 | 生命に関わる情報の遅延防止 |

### 7.4 投稿優先度

| 優先度 | 電文種別 | 投稿遅延 |
|:-------|:---------|:---------|
| P0（即座） | 大津波警報、南海トラフ臨時情報 | 0秒（キューをバイパス） |
| P1（最優先） | 津波警報・注意報、特別警報、噴火速報 | 最大5秒 |
| P2（優先） | 震度5以上の地震、土砂災害警戒情報 | 最大15秒 |
| P3（通常） | 震度3-4の地震、気象警報、竜巻注意 | 最大30秒 |
| P4（低優先） | 震度1-2の地震、注意報、降灰予報 | 最大60秒 |

### 7.5 セッション管理

ログインのレート制限は非常に厳しい（IPベースで1日数十回程度）ため、セッショントークンをファイルに永続化する。

```typescript
// セッション永続化の方針
// 1. 初回: agent.login() → セッション情報をファイル保存
// 2. 以降: agent.resumeSession() でトークン復元
// 3. トークン期限切れ: refreshSession → 再保存
// 4. refresh失敗時のみ: 再login（1日1回まで）
```

### 8.3 エラーハンドリング

| エラー | 対処 |
|:-------|:-----|
| JMAフィード取得失敗 | 3回リトライ（10秒間隔）→ 失敗ログ → 次のポーリングで再試行 |
| 詳細XML取得失敗 | entry.contentでの簡易投稿にフォールバック |
| Bluesky投稿失敗（429） | 指数バックオフ（30秒→60秒→120秒） |
| Bluesky投稿失敗（5xx） | 3回リトライ → キューに戻して後で再試行 |
| セッション期限切れ | refreshSession → 失敗時login（1日1回制限） |
| プロセスクラッシュ | Fly.io auto-restart + 起動時に長期フィードでキャッチアップ |

## 9. ホスティング

### 9.1 フェーズ別ホスティング戦略

| フェーズ | 本番環境 | 開発・テスト環境 |
|:---------|:---------|:----------------|
| Phase 1 | Fly.io 東京リージョン | Raspberry Pi（自宅） |
| Phase 2以降 | Fly.io（スケールアップ検討） | 同左 |
| 将来構想 | Fly.io + Pi バックアップ二重構成 | — |

選定根拠：
災害情報Botは大規模災害時こそ最も必要とされる。しかし大規模災害時には自宅環境（停電・回線断）が最も脆弱になる。2018年北海道胆振東部地震では全域ブラックアウトが約45時間継続した実績がある。月額約¥330のコストで災害時の可用性を確保できるため、Phase 1からクラウドホスティングを採用する。
